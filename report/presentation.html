Scale-Up vs Scale-Out

Scale-Up

  • One powerful machine
  • Large RAM, many CPU cores
  • No network overhead
  • Simple to operate
  • Example: HPC fat node, workstation

Scale-Out

  • Many commodity machines
  • Distributed memory and compute
  • Fault-tolerant, elastic
  • Higher operational complexity
  • Example: AWS EMR cluster

Main Question


Can a single node with enough RAM match a distributed cluster for analytical SQL workloads?

Modern servers offer lots of RAM. If data fits in memory, is the complexity of a cluster even necessary?

How We Compare

  • Run the same 6 SQL queries on both engines
  • Each query executed 3 warm runs (1 cold discarded)
  • Measure: execution time (mean, min, max)
  • We compute speedup = Spark time / DuckDB time

Expected Results

  • Spark expected to win on large full-table scans - more CPU cores (48 vs 8)
  • DuckDB expected to win on complex joins - no shuffle overhead
  • At 100 GB with equal RAM, results should be close overall
  • We expected Spark to have a clear edge - the results surprised us

When Spark Wins

Query DuckDB Spark Reason
Q6 12.04 s 3.07 s Simple filter + agg - 48 vCPU and AQE dominate
Q1 18.49 s 9.35 s Full 68 GB scan - raw CPU parallelism wins
Q10 25.17 s 14.01 s 4-table JOIN distributed across cluster

Pattern: Spark excels at large full-table scans and simple aggregations where 48 vCPU matter most.

When DuckDB Wins

Query DuckDB Spark Reason
Q21 31.30 s 55.36 s EXISTS/NOT EXISTS → local semi/anti-join, no shuffle
Q5 17.20 s 26.79 s 6-table JOIN - dimensions cached in local RAM
Q3 15.59 s 16.16 s Effectively a tie (<4% difference)

Pattern: DuckDB excels at complex joins and correlated subqueries - zero network overhead.

Conclusions

  • DuckDB and Spark are evenly matched at 100 GB with equal RAM (3 wins each) - DuckDB tied a cluster with 6× fewer cores (8 vs 48 vCPU)
  • Scale-up is simpler, cheaper, and faster to set up - no cluster management
  • Scale-out pays off when data exceeds node RAM, or when fault-tolerance / streaming is needed
  • Recommendation: start with DuckDB on a single node; move to Spark only when you hit a need above that DuckDB can't meet
